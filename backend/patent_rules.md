# 專利類別心智圖分析準則 (Patent Mind Map Rules)

1. **資料讀取**：必須完整讀取上傳之 Excel 或 PDF 內容，不可遺漏任何專利條目。
2. **應用領域分類**：
   - 預設分類標籤數量：3 至 7 個。
   - 邏輯：依據全局專利件數與"應用領域"欄位內容，並遵循預設分類標籤數量範圍，進行適當標籤命名與分類統計，計算落入各標籤不同公開公告號的專利件數。
3. **技術1階分類**：
   - 預設分類標籤數量：3 至 5 個。
   - 邏輯：依據全局專利件數與"AI技術簡述"、"技術特徵手段"及"解決的技術問題或技術效益"欄位內容（語意特徵），並遵循預設分類標籤數量範圍，進行適當標籤命名與分類統計，計算落入各標籤不同公開公告號的專利件數。
4. **技術2階分類 (依附於技術1階)**：
   - 預設分類標籤數量：3 至 7 個。
   - 邏輯：依據落入相同技術1階的專利其"AI技術簡述"、"技術特徵手段"及"解決的技術問題或技術效益"欄位內容，進行適當技術2階的標籤命名與分類統計，且每個技術1階節點下的子節點數量限制在 3 至 7 個範圍內，計算落入各技術2階標籤不同公開公告號的專利件數。
5. **技術3階分類 (依附於技術2階)**：
   - 預設分類標籤數量：3 至 5 個。
   - 邏輯：依據落入相同技術2階的專利其"AI技術簡述"、"技術特徵手段"及"解決的技術問題或技術效益"欄位內容，進行適當技術3階的標籤命名與分類統計，且每個技術2階節點下的子節點數量限制在 3 至 5 個範圍內，計算落入各技術3階標籤不同公開公告號的專利件數。   
6. **功效節點分類**：
   - 預設數量：3 至 5 個。
   - 邏輯：依據全局專利件數與"功效節點"欄位內容，並遵循預設分類標籤數量範圍，進行適當分類命名與統計，計算落入各功效節點不同公開公告號的專利件數。
7. **動態調整機制**：
   - 當使用者手動修改上述各階層分類數量設定時，AI 必須遵循設定值重新分析。
   - **核心指令**：AI 應依照設定的數量，對應調整「分類命名的廣度 (Generalization/Specialization)」，確保所有專利皆能被合理納入指定數量的分類中。若局部階層子節點超出數量限制（例如技術2階超過7個），後端應通過 AI 重新調整分類命名廣度（將子節點向上歸納合併）來滿足數量限制要求。
8. **統計準則**：心智圖上呈現的專利件數，必須是該階層所包含的不同「專利公開公告號」之去重統計總數。
9. **資料回填**：AI 僅負責分類路徑。`AI技術簡述`、`技術特徵手段`、`解決的技術問題或技術效益` 必須由後端根據專利號從原始檔案回填，嚴禁 AI 消耗 Token 生成。
10. **大數據分階處理機制 (Hybrid API Strategy)**：
    - 當專利總數不大於 50 件時，採用單次 Prompt 處理，由 AI 直接完成全域專利的分類與路徑分配。
    - 當專利總數大於 50 件時，必須切換為「兩階段處理」：先由 AI 建立全域分類目錄樹 (Global Taxonomy Tree)，再將專利分批映射對齊至該樹中，以避免 Token 上限與上下文遺失。
11. **多重路徑扁平化處理 (Path Duplication)**：
    - 若單件專利擁有多條分類路徑，後端必須為每條獨立路徑複製產生對應的專利紀錄。此舉確保前端在渲染嵌套樹狀元件時，不會產生錯誤的跨父節點映射（Cross-product bug），且前端統計時仍能依據「專利公開公告號」正確去重。
12. **根節點標籤命名規範 (Root Node `summary_title`)**：
    - `summary_title` 必須是**完整且有意義的技術主題名詞短語**，用以概括本批專利的核心技術領域。
    - 命名應簡潔（建議 8 至 16 個字），以名詞或名詞短語結尾，例如：「光子積體電路技術與應用」、「半導體先進封裝技術」、「車用感測與自動駕駛系統」。
    - **嚴禁**輸出截斷字串、孤立字符、或以動詞/助詞結尾的不完整短語（例如：「…技術與應用分」、「…技術之研究」均為不合格輸出）。
